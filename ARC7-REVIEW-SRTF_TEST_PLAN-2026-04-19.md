## Major Issues (Should Address)

### M-1: Hardcoded State Dependencies
- **Severity:** Major
- **Source:** The Architect, The Naysayer
- **Description:** The test plan relies heavily on specific commands (`git pull --rebase`, `bd dolt push`, `./scripts/bootstrap.sh`) and exact file paths (`.neurostrata/docs/NeuroStrata MemorySpace.canvas`). If the underlying tooling changes, the tests will become brittle and fail despite correct agent behavior.
- **Recommendation:** Abstract the assertions. Instead of asserting exact bash commands, assert the resulting system state (e.g., verifying the remote git branch matches the local branch, rather than parsing the bash trace for `git push`).

---

## Minor Suggestions (Nice to Have)

### MIN-1: Test Environment Isolation
- **Severity:** Minor
- **Source:** Security Sentinel
- **Description:** Pillar 4 mentions running `git push` and `bd dolt push`. In a test environment, these operations must be strictly mocked or pointed to an isolated test remote to avoid polluting the production database or repository.
- **Recommendation:** Explicitly document the setup of isolated mock remotes and LanceDB instances for the Podman container in the `train_srtf.py` mechanics section.

### MIN-2: Edge Case Coverage for Failures
- **Severity:** Minor
- **Source:** The Naysayer
- **Description:** The plan assumes the agent succeeds in its tasks. It does not explicitly test how the agent handles tool failures or context exhaustion.
- **Recommendation:** Introduce negative test cases (e.g., simulate a LanceDB connection failure or `git push` rejection) to evaluate the agent's recovery mechanics.

- **Source:** Product Visionary
- **Description:** The evaluation mechanics list four scores (Recall, Coordination, Synthesis, Commit). It is unclear if these are weighted equally or if certain failures (e.g., missing the Habitual Memory Commit) result in an immediate fail.
